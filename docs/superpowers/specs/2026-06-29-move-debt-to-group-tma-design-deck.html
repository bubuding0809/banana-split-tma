Users can see & undo transfers in-app, but can only create from the command line.

What create-transfer does

Moves an outstanding debtor → creditor debt from a source chat to a target chat. No spending logged in either group.

  • Source ≠ target · debtor ≠ creditor
  • Creator + debtor + creditor all in both chats
  • Debtor must already owe ≥ amount in source solvency, advisory-locked
  • Both groups get a Telegram notice — already built

Row → parameters

The tapped row fixes everything:

FieldFrom the row
source chat / currencyrow's chat & currency
amountabs(nativeNet) — full only
debtor / creditorsign of nativeNet
  • net < 0 caller owes → debtor=caller, creditor=counterparty
  • net > 0 they owe → debtor=counterparty, creditor=caller
  • zero-net row → not transferable, stays inert

Mutation wiring

createTransfer.mutate({
  sourceChatId, targetChatId,
  debtorId, creditorId,
  amount,                 // row nativeNet, abs
  currency,               // row currency
  description: undefined, // no note field
})

onSuccess: haptic, close, invalidate the same caches delete already does + getMyCounterpartyBalances.

onError: haptic + Snackbar with backend message (solvency / membership).

New backend query

expenseShare.getEligibleTransferTargets

// in: { counterpartyUserId, sourceChatId }
db.chat.findMany({
  where: {
    AND: [
      { members: { some: { id: callerId } } },
      { members: { some: { id: counterpartyId } } },
    ],
    id: { not: sourceChatId },
  },
  select: { id: true, title: true },
})
// out: [{ chatId, chatTitle }]

Membership-only filter. Solvency stays server-side in createTransfer — backend remains source of truth.
