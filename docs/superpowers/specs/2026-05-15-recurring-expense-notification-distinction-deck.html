Trigger flow

Manual and recurring paths converge on one handler — that's the seam.

Manual:    user submit
        → createExpenseHandler
        → sendExpenseNotificationMessageHandler   ◀── seam

Recurring: EventBridge Scheduler
        → RecurringExpenseLambda (HMAC)
        → POST /recurring-expense-tick
        → createExpenseHandler  (recurringTemplateId set)
        → sendExpenseNotificationMessageHandler   ◀── same seam

Expense.recurringTemplateId non-null ⇒ auto-created. Single source of truth.

Deep-link discipline

load-bearing Encoder is shared by e, s, p, c — counterparty links and snapshot shares depend on it.

  • TDD. Lock existing entity round-trips with tests first.
  • Then add a failing rt round-trip test.
  • Then change the encoder — green new test, green old tests.
  • Same for the TMA start_param parser: existing branches locked first.
  • UAT must include taps on a pre-existing View Expense, snapshot share, and counterparty deep link. Any regression = rollback.

The new rt entity is additive only. Existing wire format is frozen.

Plumbing

+ recurringTemplateId on input schema.

+ 2nd button in keyboard builder.

Edit-expense path

Re-read expense.recurringTemplateId from DB; pass the same way. Footer & button persist across edits naturally.

Data model

No schema change. Every signal already exists.

FieldRole

RecurringExpenseTemplate.statusNot consulted — edit screen handles read-only state

Edge cases

  • deleted template Button still renders; edit-recurring route shows existing not-found state.
  • edit flow Body is rebuilt each edit — footer reflects current row state.
  • first occurrence Template's initial materialised expense also gets the footer + button. Desired — flags the row as recurring from day one.
  • manual → recurring later No such flow today. If added, next edit-render picks it up automatically.

UAT

  • Create recurring expense → first notification has footer + 2 buttons.
  • aws lambda invoke the scheduler → second occurrence's notification matches.
  • Tap View Schedule → lands on edit-recurring screen for correct template.
  • Edit the recurring-origin expense in TMA → updated notification preserves both.
  • Create a manual expense → only one button, no footer (regression check).

Manual lambda fire required between create and edit (see feedback_recurring_uat_lambda_fire).
