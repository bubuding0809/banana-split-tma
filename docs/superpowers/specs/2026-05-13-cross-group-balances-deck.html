Cross-Group Balances

Person-first view in the personal-chat TMA

Design · 2026-05-13 v1 spec

bubu (@bubuding0809)

The problem

Today

Balances live inside each group. 5 groups = 5 places to check.

Author manages it via CLI + an agent. not portable

Personal-chat "Groups" tab is a stub: UserBalancesTab.tsx

What users want

"How much does this person owe me, total, across every group we share?"

"One transfer to clear it all."

Goals · non-goals

v1 goals

  • One signed number per counterparty in user's base currency
  • Drill-down per (group, currency)
  • One-tap full settle, native currency, all buckets
  • Nudge via bot DM · settle notification via bot DM
  • CLI parity

Not in v1

  • Partial settle (per-group screens still do this)
  • CrossGroupSettlement parent record
  • Rate snapshots — conversion is live
  • External-payment notes
  • Bulk multi-counterparty settle

Mental model

Top-level list = people with non-zero open balance across all shared groups.

Each row = one signed amount in user's base currency.

Tap row → bottom sheet with per-(group, currency) breakdown + actions.

Sorted by |totalBaseNet| desc. Empty state when no balances.

Behaviour

Each line keeps its own direction — Sean can owe in Bali but you can owe in Dinner. Aggregate is the net.

Settle confirm recomputes the SGD total at tap time (live rate may have shifted vs. when the sheet opened).

Nudge disabled if counterparty hasn't started the bot, or within 24h cooldown.

How we use it

New balances procedure calls getMultipleRates once per request, then converts all (chat, currency) buckets in memory.

TTL inherited from CURRENCY_CACHE_TTL.

No new FX util

Settle flow

  1. Recompute balance from DB inside transaction (never trust client).
  2. For each non-zero (chatId, currency) bucket → one Settlement row in native currency, correct direction.
  3. Best-effort bot DM to counterparty: 
    "Bubu just settled with you across 3 groups.
Approx S$99.42. Bali: $40.00 USD · Dinner: A$30.00 · Roommates: ¥100.00."
  4. Return refreshed (zero) balance.

per-group ledgers stay perfectly accurate — settlement is always in native amount.

Nudge flow

  1. Rate-limit check: Redis key nudge:<a>:<b> · TTL 86400s. 24h
  2. Recompute live balance. Bail if caller not net-owed.
  3. Best-effort bot DM to debtor: 
    "Bubu is awaiting settlement.
You owe ≈ S$99.42 across 3 groups."
  4. If debtor hasn't started bot → COUNTERPARTY_BOT_NOT_STARTED (UI pre-disables button).
  5. Set Redis key with 24h TTL on success.

Nudge is UI-only — intentionally not exposed via CLI.

Edge cases

FX API down, no cachenative amounts only · banner
Mixed direction (owe + owed same person)each bucket settles in its real direction
Counterparty hasn't started botnudge disabled · settle DM silently skipped
Rate moves between sheet open and confirmrecompute at confirm; native settlement unaffected
Concurrent expense before settletx recomputes from DB — always correct
Currency not in DBomit from sum, inline warning

Acceptance

  • 3+ groups, mixed currencies → one row per counterparty, signed SGD total, sorted by magnitude.
  • Tap → per-(group, currency) breakdown + rate timestamp.
  • Mark all settled → one Settlement per non-zero bucket, native, correct direction, counterparty disappears.
  • Counterparty receives one bot DM summarizing.
  • Nudge sends DM · 2nd within 24h returns rate-limit + button shows countdown.
  • Base currency persists across sessions; in-view picker is a session override.
  • CLI parity verified.
