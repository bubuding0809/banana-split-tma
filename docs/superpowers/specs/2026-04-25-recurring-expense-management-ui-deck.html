Design spec · 2026-04-25

Recurring Expense
Management UI

Closing the management gap. The backend exposes list / get / update / cancel for recurring expense templates, but the TMA only renders the bare list. Designing four surfaces to view, edit, and delete — sharing the existing expense Section/Cell language end to end.

Status: Approved Author: bubu (with Claude Opus 4.7) Surfaces: 4 Backend changes: 0
The problem

Backend is ready. UI is bare.

What exists today

  • expense.recurring.list — active templates per chat
  • expense.recurring.get — single template
  • expense.recurring.update — partial mutate (schedule + amount + description)
  • expense.recurring.cancel — soft-delete + AWS schedule cleanup

What the user can do today

  • See a flat list of templates with no context
  • Tap a row → nothing happens
  • No way to edit the schedule
  • No way to cancel/delete from the app
Design constraints

Reuse over invent. Schedule-only edit. No backend changes.

Reuse, don't reinvent

All four surfaces mirror existing expense Section/Cell shapes. Users see one visual language for the same domain object.

Schedule-only mutation

Form edits frequency / interval / weekdays / endDate. Description, amount, payer, splits, category, currency are immutable — cancel & recreate to change.

Zero backend changes

Every interaction maps to an existing tRPC procedure. No schema changes, no new mutations.

Out of scope (v1)

✗ Per-occurrence editing (calendar-style "this vs. all future")
✗ Pause / resume (backend doesn't support)
✗ Bulk operations
✗ Editing payer / splits / category / currency
✗ Surfacing recurring schedules in chat-list summary

Architecture

One refactor, two new components, two augmentations

Pure refactor

AmountFormStep.tsx:419-590 → extract Repeat Cell + End Date Cell + RecurrencePickerSheet wiring into shared RepeatAndEndDateSection component. No behavior change. Existing 27 tests must keep passing.

NEW components

RecurringExpenseDetailsModal (presentation only)
EditRecurringSchedulePage (page, mounts RepeatAndEndDateSection)

Augmented

RecurringTemplatesList — visual upgrade, deeplink param
ExpenseDetailsModal — conditional Schedule section

tRPC contract

Nothing new. list, get, update, cancel all already shipped with the recurring feature. Client computes "Next: " via existing nextOccurrenceAfter.

Routing

One new route — that's it

PathComponentStatus
/chat/$chatId/recurring-expensesRecurringTemplatesListexisting — visual upgrade
/chat/$chatId/edit-recurring/$templateIdEditRecurringSchedulePageNEW route

Management flow

Settings → Recurring expenses → Tap row → RecurringExpenseDetailsModal → Tap pencil → Edit page

From transactions, the augmented modal is read-only — no cross-modal navigation. Per-occurrence edit and template management are intentionally separated by entry path.

Resilience

Three failure modes, three responses

Save fails

Network or AWS UpdateScheduleCommand error. Toast + keep form dirty so user can retry. tRPC mutation already returns the AWS error verbatim — display as-is.

Cancel: AWS fails, DB succeeds

Per cancel.ts:47-58, template is marked CANCELED in DB even if AWS fails. List filters to ACTIVE so the row disappears immediately. Schedule may keep firing for ≤24h but tick endpoint rejects CANCELED templates. No user-visible effect.

Race: deleted on another device

tRPC throws NOT_FOUND on the cancel mutation. Catch in onError: toast "This recurring expense was already deleted" → close modal.

Verification

Tests + manual UAT

Unit (must keep passing)

The RepeatAndEndDateSection extraction is a pure refactor. Existing suites must continue to pass:

  • recurrencePresets.test.ts — 27 tests
  • AddExpenseForm.type.test.ts — 12 tests

Component smoke

RecurringExpenseDetailsModal renders the right Sections given:

  • Mock template with no shares vs. with shares
  • Mock template with endDate vs. without
  • Mock template with each frequency (Daily / Weekly / Monthly / Yearly / Custom)

Manual UAT (full-environment lifecycle)

One cycle each for view, edit-and-save, edit-and-discard, delete. After each: verify DB state via Supabase MCP and AWS Scheduler state via aws scheduler get-schedule. Per the full-environment-UAT loop preference.

In one screen

The whole design in 7 bullets

  • List page rows look like ChatExpenseCell — same visual language as transactions
  • Tap row → bottom-sheet modal mirroring ExpenseDetailsModal sections + a new Schedule section
  • Modal pencil → focused /edit-recurring/$templateId page (single section, schedule-only)
  • Modal & edit page share Telegram bottom bar pattern: Save + Delete
  • Existing transaction modal grows a read-only "↻ Recurring schedule" section so users see at a glance "this expense is recurring" — no deeplink, scope stays per-occurrence
  • One refactor (extract RepeatAndEndDateSection), two new components, two augmentations, zero backend changes
  • Schedule-only mutation in v1; everything else immutable until users ask for more
