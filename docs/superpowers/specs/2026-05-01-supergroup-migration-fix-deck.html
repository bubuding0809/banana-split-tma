Supergroup migration reliability fix

Replace the broken in-memory race guard with a self-correcting design
that tolerates dropped events, cross-instance routing, and concurrent triggers.

2026-05-01 · design spec

Three structural flaws to fix

A single event side

Bot listens only to migrate_to_chat_id on the old chat. Telegram's symmetric event on the new supergroup is ignored.

B in-memory dedup

migratedChatIds Set is per-instance heap. Cross-instance webhook routing on Vercel makes the 5s race-condition guard ineffective.

C silent catch

Migration errors disappear into a single console line. No alerts, no rethrow, short retention.

Each flaw alone can break a migration. Labour day was likely flaw A; flaw B is a ticking bomb regardless.

Solution: 5 components landing together

  1. Symmetric Telegram listener — add message:migrate_from_chat_id handler, both call shared runMigration helper.
  2. Idempotent migrateChat + advisory lock — Postgres pg_advisory_xact_lock on newChatId; re-read state inside lock; idempotent no-op when old chat already gone.
  3. "Old wins" race-branch merge — extend race-branch to move all chat-scoped tables; replace new chat's defaults with old's.
  4. Message dedup via Chat.migratedFromChatId — new nullable column distinguishes "fresh add" from "migrated"; my_chat_member uses it to skip welcome on migrated chats.
  5. Stop swallowing errors — let migrate errors propagate to grammy's bot.catch.

Component 2: Idempotent migrateChat


await db.$transaction(async (tx) => {
  // Serializes against any concurrent migrate for this newChatId.
  // Auto-released on commit/rollback. Cross-instance safe.
  await tx.$executeRaw`SELECT pg_advisory_xact_lock(${newChatId}::bigint)`;

  const oldChat = await tx.chat.findUnique({ where: { id: oldChatId } });
  if (!oldChat) return { migrated: false };  // ← idempotent no-op

  const newChat = await tx.chat.findUnique({ where: { id: newChatId } });
  if (newChat) {
    // race-branch: explicit merge with "old wins" policy
    return { migrated: true };
  } else {
    // raw SQL UPDATE on Chat.id; cascade does the rest
    return { migrated: true };
  }
});
Returning { migrated: false } for the no-op case is what lets us de-duplicate the user-visible MIGRATION_MESSAGE_GROUP.

Component 4: differentiating new vs migrated


model Chat {
  id                  BigInt   @id
  // ... existing fields
  migratedFromChatId  BigInt?    // ← new
}

my_chat_member rules

  • Chat exists, migratedFromChatId set → migrated chat → skip welcome
  • Chat exists, migratedFromChatId null → re-add → skip welcome
  • Chat doesn't exist → fresh add → createChat + send welcome

runMigration rules

  • migrateChat returns { migrated: true } → send MIGRATION_MESSAGE_GROUP
  • migrateChat returns { migrated: false } → no message (idempotent path)

Out of scope

  • deferred Admin slash command /migrate <oldId> for manual recovery
  • deferred Alerting on migration failure (Sentry/Slack) — current bot.catch + Vercel logs is the agreed minimum
  • deferred Heuristic post-hoc migration via member overlap
  • accepted The "welcome + migration" edge case when my_chat_member fires first; closing it would need a DB-poll, judged not worth the latency

Testing

Handler-level

  • migrated: true for fresh-Branch-B and race-branch; migrated: false when old chat already gone
  • Race-branch moves all 7 chat-scoped tables; sets migratedFromChatId
  • createChat returns existing row on unique-conflict; schedule handler no-ops if exists

Concurrency

  • Two parallel migrateChat calls for same pair → one does work, one no-ops; no DB inconsistency

Manual UAT (Telegram)

  • Real upgrade with expenses + categories + snapshot — all carry over, exactly one migration message
  • Bot offline during upgrade, online after — migration still completes via new-side event
  • Direct fresh-supergroup creation — welcome only, no migration message
  • Kick + re-add — no welcome, no duplicate seeding
