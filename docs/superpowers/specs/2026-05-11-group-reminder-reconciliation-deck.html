DB ↔ AWS scheduler reconciliation

Weekly diff · self-heal missing schedules · log orphans & drift

Design spec · 2026-05-11 · group-reminder pipeline

The bug that triggered this

Group "LADS 2026" (-1003625813655) had been in the DB since 2026-02-20. It never received a Sunday 9pm SGT summary — twelve weeks in a row, silently skipped.

  • root EventBridge schedule group-reminder-1003625813655 never existed in AWS
  • silent createChat's finally block catches and swallows the failure
  • reactive PR #262's toggle-on upsert only fires when a member opens settings — too late for LADS 2026
  • found only because someone opened settings on 2026-05-11 — already missed Sunday
No Axiom event. No Vercel error log. No DB indicator. The chat exists; the schedule doesn't; the bot never fires.

Goals

  1. Every DB chat has an EventBridge schedule before the next Sunday 21:00 SGT fire window — even if createChat silently failed.
  2. Each run emits one summary event (reconciliation.run.complete) with bucket counts + per-chat events for anything weird.
  3. Diff logic is pure — unit-testable without AWS or DB mocks.
  4. Orphans and drift are logged only. Deletion / repair needs human review.

The three buckets

missing

DB chat exists, no schedule.

Action: auto-create with createChat defaults (Sun 21:00, SGT, enabled).

Event: reconciliation.schedule.created

orphan

Schedule exists, no DB chat.

Action: log only. Most are group→supergroup migration leftovers.

Event: reconciliation.schedule.orphaned

drift

Disabled, wrong cron, wrong tz, or wrong ARN.

Action: log only. Most drift is user-intentional.

Event: reconciliation.schedule.drift

Pure diff function

export function reconcileSchedules(
  chats: { id: bigint; type: ChatType }[],
  schedules: {
    name: string; state: "ENABLED" | "DISABLED";
    scheduleExpression: string; timezone: string; targetArn?: string;
  }[],
  expectedArn: string,
): {
  missing: { chatId: string }[];
  orphan:  { scheduleName: string }[];
  drift:   { chatId: string; scheduleName: string;
             reasons: ("disabled"|"wrong_cron"|"wrong_tz"|"wrong_arn")[] }[];
  ok:      { chatId: string }[];
}

No AWS client. No DB client. No logger. Pass two lists, get four buckets. Eight unit-test cases cover the matrix.

Pagination — ListSchedules caps at 100

Live env has 230 schedules; we always paginate.

async function listAllGroupReminderSchedules() {
  const all = [];
  let nextToken: string | undefined;
  do {
    const resp = await client.send(new ListSchedulesCommand({
      GroupName: "default",
      NamePrefix: "group-reminder-",
      MaxResults: 100,
      NextToken: nextToken,
    }));
    for (const s of resp.Schedules ?? []) {
      all.push(await fetchScheduleDetails(client, s.Name!));
    }
    nextToken = resp.NextToken;
  } while (nextToken);
  return all;
}

ListSchedules returns summary only — need GetSchedule per item for cron/tz/ARN. Batch in parallel chunks of 10.

One-off orphan cleanup — lands first

The recurring job assumes the orphan list is small. Without cleanup, every Saturday run spams 130 reconciliation.schedule.orphaned warns for cases already triaged.

  • script apps/admin/scripts/reconcile-orphans-oneoff.ts
  • Parses chatId from each group-reminder-N name, checks both ± candidates against chat table
  • Cross-references the supergroup migration log (migratedFromId) to classify as migration_leftover / dev_test / unknown
  • Emits CSV with recommendation; operator reviews; --apply deletes only the agreed rows
  • human-in-loop one-shot drain, not a recurring auto-delete

One monitor, two conditions: create_failed > 0 OR no run.complete by Sat 10:00 SGT.

Failure modes

  • list fails mid-pagination top-catch logs run.failed; skip create pass; re-run manually. Don't double-create on a partial list.
  • one createSchedule fails log create.failed, loop continues. Chat stays missing until next run or settings-toggle.
  • one getSchedule fails skip that one; drift detection deferred. Log get.failed.
  • route times out Axiom monitor fires on missing run.complete. Operator re-runs.
  • retry policy none in-route. SDK already retries individual calls; full-run retry would mask sustained outages.
