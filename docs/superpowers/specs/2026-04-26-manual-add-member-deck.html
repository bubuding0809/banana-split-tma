Design spec · 2026-04-26

Manual Add Member
Wire TMA → Bot

Wire the placeholder "Add Member" button in the Members sub-page to the existing bot-side /start ADD_MEMBER<chatId> flow. Most of the work is already done — the bot already runs a multi-select requestUsers picker. This spec finishes the wiring and adds three small bot-side improvements.

Status: Awaiting user review Author: bubu (with Claude Opus 4.7) New tRPC: 0 New routes: 0 Schema changes: 0
Why this exists

Telegram bots can't list group members

Unless the bot is a group admin, it can't see who's in the chat. Banana Splitz registers users by having them open the TMA — but when the bot posts that prompt, some users miss it and never get added.

The fix: let users already in the group manually onboard the others. Tap "Add Member" in Settings → pick from your Telegram contacts → done.

Today (placeholder)

"Add Member" button opens a sheet that says "This flow isn't wired up yet — coming soon." Nothing happens.

After this spec

Tap → brief explainer sheet → "Open bot DM" → Telegram contact picker → multi-select up to 10 → bot adds them, optionally DMs them, sends you back to the app.

Decision · brainstorm output

Why bot deep-link, not in-TMA user search?

An in-TMA search across the User table was on the table. It loses on the headline use case.

Bot DM requestUsers In-TMA search
Works for users not yet in the User table ✓ Telegram is the source of truth ✗ Requires prior bot interaction
Already implemented ✓ apps/bot/src/features/user.ts ✗ New procedures + new page
Native Telegram contact picker ✓ ✗ Custom search list
Stays inside the TMA ✗ ~3s round-trip ✓

Headline use case is "add people who haven't opened the TMA yet" — exactly the population that won't be in the User table and therefore can't be found via in-TMA search. Bot deep-link is the only path that covers it.

User flow · happy path

Six taps, one round-trip

  1. From the chat in TMA, tap Settings → Members → Add Member.
  2. Brief sheet explains the flow, primary CTA is "Open bot DM". Tap it.
  3. TMA closes, Telegram opens the bot DM with the contact picker keyboard already mounted (requestUsers, multi-select up to 10, names + usernames).
  4. Pick the people you want to add, tap Send. Telegram sends their user_id + first_name + last_name + username to the bot.
  5. Bot adds them (creating User rows for first-timers), optionally DMs each one with an "Open app" button, and replies in your DM with "Added: …" + an inline button to return to the chat in the TMA.
  6. Tap the inline button → back in the TMA. Members sub-page auto-refreshes via visibilitychange listener.

Surface 3 · Bot

Three small bot-side improvements

Membership guard

Before processing users_shared, call getChatMember(groupId, ctx.from.id). If status is left / kicked, reply with "you can only add members to groups you're a member of" and abort.

Why: today, anyone who knows a chat ID can craft the deep link and inject members. Low risk in practice; still a real gap. Fail-closed on errors.

Inline keyboard back to TMA

After the success message, attach an inline keyboard with "Open <chat> in app". Builds the URL via ChatUtils.createMiniAppUrl + createChatContext — already used elsewhere.

Why: closes the loop without forcing the user to swipe back through Telegram navigation.

DM each added user

Per-user, after a successful add: sendMessage(userId, "you were added by …") with an Open App button. Wrap in try / catch — Forbidden errors (target never DM'd the bot) are expected and silently swallowed.

Why: the headline problem is users missing the original prompt. A direct DM closes the loop for repeat-bot users; first-timers fall back to the existing flow.

Edge cases

What can go wrong, and what we do

User shares themselves or someone already in chatchat.addMember returns CONFLICT → counted in failed list. Cosmetic only.
User shares a Telegram bot accountTelegram's requestUsers defaults to user_is_bot: false. Filter happens at picker level — we never see bots.
Bot can't fetch chat title (private group)Existing fallback "the group" applies. CTA reads "Open the group in app".
Requester left the group between deep-link tap and contact shareMembership guard rejects with ADD_MEMBER_NOT_A_MEMBER. Session cleared.
Bot was removed from the groupgetChatMember throws → caught and treated as "not a member" (fail-closed).
Bot can't DM an added user (target never DM'd bot)Forbidden caught and swallowed. Success list still includes them.
Session expires between deep-link and contact shareExisting handler already says "Session expired. Please start the add member process again."
TMA is in background while user adds via botvisibilitychange listener invalidates chat.listMembers on return. List updates without manual refresh.

What we're not changing

  • No new tRPC procedures — chat.addMember already does the right thing for the bot's authenticated calls
  • No schema changes — no new tables, no new columns
  • No new TMA routes — same /_tma/chat/$chatId/settings tree
  • No bot deep-link protocol changes — re-using ADD_MEMBER<chatId> start arg already implemented in userFeature.command("start")
Verification

Tests + manual UAT

Component smoke

  • AddMemberSheet: renders new copy, primary CTA calls openTelegramLink with the expected URL, secondary CTA closes the sheet
  • MembersSubPage: passes chatId prop; visibilitychange listener invalidates exactly once per visible→hidden→visible cycle

Subagent UAT (programmable)

  • Pre-flow DB audit: User rows for test targets (mix of pre-existing + fresh)
  • Post-flow DB audit: _ChatMembers rows for each, User rows created for new ones
  • Cleanup: subagent removes any test rows it created

Manual UAT — full lifecycle (per project convention)

  1. Happy path: TMA → Settings → Members → Add Member → sheet → Open bot → bot DM → pick 2 contacts → success message with deep-link → tap → back in TMA → Members shows new entries
  2. Notification path: if added user has DM'd bot before, they receive the "you were added" message with Open App button
  3. Notification swallow: add a fresh user — no error surfaces, success list still includes them
  4. Membership guard: craft ?start=ADD_MEMBER<chatId-not-yours> link, tap → bot rejects with the new error string
  5. Cancel: "❌ Cancel" keyboard button → session cleared, keyboard removed, fresh re-tap from TMA still works
  6. TMA refresh: background TMA + bot flow → return to TMA → members list auto-updates via visibility-change listener
In one screen

The whole spec in 5 bullets

  • TMA wiring — replace the "coming soon" placeholder in AddMemberSheet.tsx with a real openTelegramLink('?start=ADD_MEMBER') CTA. Same SDK call already used at NewUserPage.tsx:31.
  • Visibility refetch — when the user returns from the bot DM, MembersSubPage invalidates chat.listMembers via a visibilitychange listener. List updates automatically.
  • Membership guard — bot's users_shared handler verifies the requester is actually in the group via getChatMember before adding anyone. Fail-closed on errors. Closes a low-risk but real gap.
  • Better success message — append an inline-keyboard "Open <chat> in app" button using the existing ChatUtils.createMiniAppUrl helper.
  • Notify each added user — best-effort DM from the bot with an Open App button. Silently swallows Forbidden for first-time targets.
