Avatars Overhaul
Real photos, no leaks

Three avatar surfaces, one PR. Replace per-user emoji with real Telegram photos. Close the bot-token leak that ships in production today on group photos. Use initData directly for the user's own face — instant, free.

Why this exists

Two token leaks, one fix shape

Telegram's Bot API only ever issues file URLs of this shape:

What Telegram returns
https://api.telegram.org/file/bot7234567890:AA…secret…/photos/file_42.jpg

User-photo attempt

The previous attempt at user avatars returned this URL straight to the client. Disabled when the leak was discovered. Stub at getUserProfilePhotoUrl.ts returns null.

Group-photo (live in prod)

routers/telegram/getChat.ts calls teleBot.getFileLink(big_file_id) and returns the URL to the client. Anyone with DevTools open on a group page reads the bot token right now.

Why bundle the fixes: three PRs would mean a window where the group leak coexists with the new user proxy. One PR closes everything in the same deploy.
Approach matrix

One source per surface

For self, the initData CDN URL skips the lambda entirely on first paint. If photo_url is undefined (privacy-hidden) the component falls through to the proxy, which 404s and lands on emoji. Defense in depth.

The shared infrastructure

The two proxy routes are nearly identical: same auth (TMA initData), same cache headers, same fallback shape. Differ only in (a) authz check — shared-chat vs chat-membership — and (b) the Telegram API call shape. ~150 LOC total across both.

Surface 1 · Self avatar

Free win from initData

Telegram already hands us a CDN URL for the current user. Bot API 8.0 (Nov 2024) made photo_url available to all Mini Apps regardless of entry point.

URL shape
https://t.me/i/userpic/<size>/<hash>.jpeg
  • No bot token in path
  • Hotlink-friendly Telegram CDN
  • Hash-keyed = cache-immutable per upload
  • Available via useLaunchParams().initData.user.photoUrl

Caveats

  • Optional — undefined when user privacy hides photos from non-contacts. Component falls through to proxy.
  • Per-launch — initData is set when TMA opens. Photo changes mid-session won't appear until next launch. Acceptable.
  • Self only — there's no API to get other users' t.me/i/userpic URLs from the Bot API. Surface 2 covers everyone else.
Surface 2 · User proxy code

apps/lambda/api/avatar.ts

router.get("/:userId", async (req, res) => {
  // 1. Auth — TMA initData (header OR ?auth= for <img>)
  const initData =
    req.header("authorization")?.replace(/^tma /, "") ??
    (req.query.auth as string | undefined);
  if (!initData) return res.status(401).end();
  let callerId: number;
  try {
    validateInitData(initData, env.TELEGRAM_BOT_TOKEN);
    callerId = parseInitData(initData).user!.id;
  } catch { return res.status(401).end(); }

  // 2. Authz — caller and target share a chat
  const targetId = BigInt(req.params.userId);
  if (BigInt(callerId) !== targetId) {
    const shared = await prisma.chat.findFirst({
      where: {
        members: { some: { id: BigInt(callerId) } },
        AND:     { members: { some: { id: targetId } } },
      },
      select: { id: true },
    });
    if (!shared) return res.status(403).end();
  }

  // 3. Telegram fetch — token URL stays in this function
  const photos = await teleBot.getUserProfilePhotos(
    Number(targetId), 0, 1
  );
  const biggest = photos.photos[0]?.at(-1);
  if (!biggest) {
    res.setHeader("Cache-Control",
      "public, max-age=3600, s-maxage=3600");
    return res.status(404).end();
  }
  const fileLink = await teleBot.getFileLink(biggest.file_id);
  const upstream = await fetch(fileLink.toString());
  const bytes = Buffer.from(await upstream.arrayBuffer());

  res.setHeader("Content-Type", "image/jpeg");
  res.setHeader("Cache-Control",
    "public, max-age=86400, s-maxage=604800, " +
    "stale-while-revalidate=604800");
  res.status(200).send(bytes);
});
Surface 3 · Chat photo proxy

apps/lambda/api/chat-photo.ts

Same shape as user proxy. Two differences: authz (caller is a member of the chat) and fetch (getChat instead of getUserProfilePhotos).

router.get("/:chatId", async (req, res) => {
  // ...same auth pattern as avatar.ts...

  // Authz — caller is a member of the chat
  const chatId = BigInt(req.params.chatId);
  const member = await prisma.chat.findFirst({
    where: {
      id: chatId,
      members: { some: { id: BigInt(callerId) } },
    },
    select: { id: true },
  });
  if (!member) return res.status(403).end();

  // Telegram fetch
  const chat = await teleBot.getChat(Number(chatId));
  const bigFileId = chat.photo?.big_file_id;
  if (!bigFileId) {
    res.setHeader("Cache-Control",
      "public, max-age=3600, s-maxage=3600");
    return res.status(404).end();
  }
  const fileLink = await teleBot.getFileLink(bigFileId);
  const upstream = await fetch(fileLink.toString());
  const bytes = Buffer.from(await upstream.arrayBuffer());

  res.setHeader("Content-Type", "image/jpeg");
  res.setHeader("Cache-Control",
    "public, max-age=86400, s-maxage=604800, " +
    "stale-while-revalidate=604800");
  res.status(200).send(bytes);
});
Surface 1+2 · Web

ChatMemberAvatar — three layers

One component, three rendering paths chosen at runtime:

  1. If userId === currentUserId and initData has photo_url → use it directly. Telegram CDN, no backend hit.
  2. Otherwise → <img src="/api/avatar/:userId?auth=…"> proxy.
  3. On any image error → emoji via getAnimalAvatarEmoji.
const { initDataRaw, initData } = useLaunchParams();
const [failed, setFailed] = useState(false);

const src = useMemo(() => {
  if (failed) return undefined;
  // Self → initData CDN URL (no backend hit)
  if (initData?.user?.id === userId && initData.user.photoUrl) {
    return initData.user.photoUrl;
  }
  // Others → proxy
  return `${AVATAR_BASE}/${userId}?auth=${
    encodeURIComponent(initDataRaw ?? "")
  }`;
}, [failed, initData, initDataRaw, userId]);

return (
  <Avatar
    size={size} src={src}
    onError={() => setFailed(true)}
    fallbackIcon={getAnimalAvatarEmoji(String(userId))}
  />
);
Surface 3 · Web + tRPC cleanup

Closing the group leak

In GroupPage.tsx

Two <Avatar src={tChatData?.photoUrl}> usages today (lines ~345, ~376). Both swap to:

const chatPhotoSrc =
  `${CHAT_PHOTO_BASE}/${chatId}` +
  `?auth=${encodeURIComponent(initDataRaw ?? "")}`;

<Avatar size={28}
  src={chatPhotoSrc}
  onError={...} />

In routers/telegram/getChat.ts

Stop calling getFileLink entirely. Strip the photoUrl field from the response — there's no legitimate client use, and leaving it tempts a future contributor to wire the leak back in.

const { photo: _, ...rest } = chat;
return rest;

Other consumers of telegram.getChat use title/type/etc. — those still work.
