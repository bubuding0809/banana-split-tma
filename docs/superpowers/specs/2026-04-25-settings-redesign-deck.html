Design spec · 2026-04-25

Settings Redesign
Hub & Spoke

Replace the cramped, monolithic settings page with an iOS / Telegram-iOS hub-and-spoke layout — one focused sub-page per concern. Adds a Members roster (today there's no way to see who's in the chat) and required token names so users can tell their CLI from their agent.

Status: Awaiting user review Author: bubu (with Claude Opus 4.7) New routes: 6 Schema changes: 2 (name on both API key tables)
The problem

Six stacked sections, two of them named "Notifications"

A single 429-line component renders Currency, Categories, Personal Info, two Notifications sections, and a heavy Access Tokens block — all on one scrolling page. Pain points the user confirmed: too long, bad grouping, visually noisy, hard to find things.

Length

~2,800 lines across the Settings folder. The hub alone is 429.

Grouping

Phone number sits between Categories and Notifications — no theme.

Conflict

Two sections both labeled "Notifications" — chat events and recurring reminders.

Density

Access Tokens dominates the bottom with copy/setup/agent flows inline.

The pattern

iOS / Telegram-iOS hub & spoke

Top page is a clean menu. Each row owns one concern in its own sub-page. Visual language: solid colored icon squares, grouped sections with tiny uppercase headers, tap-through chevrons, current value previews on the right.

Hub & spoke ✓

Recommended. Native pattern. Each setting gets room. Preview values keep the hub informative at a glance.

Collapsible accordion

One page, collapse inline. Conflicts with the sub-pages categories already uses. Skipped.

Top-level tabs

Group / You / Advanced segmented control. Doesn't shorten any single tab. Skipped.

The hub

Group hub vs Private hub

Group chats see a Members row, member-avatar stack under the chat header, and Notifications/Reminders rows. Private chats collapse to just Personal-relevant rows.

New feature

Required token names

Today's ChatApiKey and UserApiKey rows are indistinguishable. Each gets a required name column so users can tell their CLI from their agent. Sheet on create, tap to rename, same component handles both.

Schema

  • ChatApiKey.name: String (required)
  • UserApiKey.name: String (required)
  • Two-step migration: add nullable, back-fill existing rows with "Token · MMM DD" from createdAt, then NOT NULL.
  • Migration ships before the new generate mutation, otherwise blank inserts fail.

tRPC

  • Modify apiKey.generateToken + user equivalent: now requires name: z.string().trim().min(1).max(40).
  • Add apiKey.renameToken({ chatId, tokenId, name }) + user equivalent. Same validation. Cross-chat tokenIds rejected.

UI flow

  • Create: "+ Generate new token" → bottom sheet with Name input (required, placeholder "e.g., CLI on Macbook"), Cancel + Create. Create disabled until non-empty after trim. Existing copy-key modal opens on success.
  • Rename: tap any token row → same sheet pre-filled. Adds a destructive Revoke button (existing confirm dialog) alongside Save.
  • Sheet uses Telegram-UI Modal. Single shared component, props switch between Create and Edit modes.

Rollout

Migration plan

Two-step Prisma migration so the NOT NULL constraint is safe under existing rows. Ships before any new mutation that requires a name.

Step 1 · Backfill

  • Add name column to ChatApiKey and UserApiKey as nullable.
  • Migration script back-fills with "Token · MMM DD" from each row's createdAt.
  • Verify zero null rows post-script.

Step 2 · Tighten

  • Alter both columns to NOT NULL.
  • Deploy the new generateToken + renameToken procedures with required-name validation.
  • Front-end updates land alongside (sheet UI).

If we ship the new mutation first, any blank insert fails validation but the column is still nullable for the back-fill — order matters.

Verification

Testing & UAT

Per memory: subagent-driven verification for backend / DB, manual AskUserQuestion walkthrough for UI/Telegram surfaces.

Subagent (automated)

  • generateToken rejects empty name, trims, enforces 40 char max.
  • renameToken same validation, rejects cross-chat tokenIds.
  • Migration leaves zero null names; column ends NOT NULL.
  • chat.listMembers respects chat membership; cross-chat returns empty/403.

Manual (Telegram UAT)

  • Hub renders correctly group vs private; member-avatar stack shows.
  • All 7 sub-pages reachable; back button returns to hub.
  • Members "Add Member" sheet opens; Coming-soon copy correct.
  • Currency picker: pick → hub preview updates.
  • Event alerts toggles persist; bot stops/starts notifying.
  • Recurring reminder toggle + schedule modal flow.
  • Phone add/remove via Telegram contact share.
  • Token: create with name, copy key, rename, revoke (group + private).
