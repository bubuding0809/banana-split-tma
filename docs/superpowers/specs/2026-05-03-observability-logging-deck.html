Backend observability + structured logging

pino → stdout → Axiom · request_id correlation · agent-accessible

Design spec · 2026-05-03

The pain that triggered this

Users hit "Something went wrong" intermittently when opening the TMA. We had a deep link, a chat id, and a timeframe. The investigation still took most of a session.

  • silent auth middleware catch — no log line on UNAUTHORIZED
  • silent getUser catch — Prisma errors swallowed into INTERNAL_SERVER_ERROR
  • invisible tRPC procedure errors return HTTP 200 — Vercel access logs flag nothing
  • missing no errorFormatter on tRPC init — no chance to log centrally
A 5,000-record / 1-hour vercel logs pull returned zero error or warning entries. The bug was real, the log surface was empty.

Goals

  1. Every tRPC procedure error and bot handler error produces exactly one structured log line in production.
  2. Every line carries request_id (and user_id / chat_id when known) so an agent can reconstruct one user's session from a single query.
  3. An agent in this repo can query prod logs via a documented APL recipe in AGENTS.md + a pnpm logs:trace script. No dashboard hunting.
  4. Three Axiom monitors fire on auth-failure spikes, error-code spikes, and p95 latency regressions.

Architecture

┌──────────────────────────────────────────────────────────┐
│  apps/lambda  (Express + tRPC, Vercel Function)          │
│   ├─ withRequestContext  ─► assigns request_id           │
│   ├─ withRequestLogger   ─► logs req.start / req.end     │
│   └─ tRPC adapter                                        │
│        ├─ errorFormatter ─► logs every procedure error   │
│        └─ ctx.log        ─► used inside handlers         │
│                                                          │
│  apps/bot  (Telegraf, Vercel Function)                   │
│   └─ telegraf middleware ─► logs every update / error    │
└─────────────────────────┬────────────────────────────────┘
                          │ JSON to stdout
                          ▼
              ┌─────────────────────────┐
              │ Vercel runtime          │
              │ → Axiom marketplace     │
              │   integration forwards  │
              └─────────────────────────┘

No network code in the app. Axiom outage queues in Vercel; nothing in the request path breaks.

Components

1. packages/logger

Pino factory + AsyncLocalStorage request context + Express middlewares.

2. Lambda wiring

apps/lambda/api/index.ts mounts middlewares before tRPC, error handler last.

3. tRPC errorFormatter + ctx.log

Procedure errors logged once with full context. Child logger pre-bound to request_id, procedure, user_id, chat_id.

4. Auth middleware logging

One line per validation failure mode before rethrowing as UNAUTHORIZED.

5. Bot Telegraf middleware

Per-update request_id + bot.catch for unhandled errors.

6. Reference id on TMA error screen

Three-line UX. Users can paste the id back to the agent.

Component 7 — agent access

MCP-only. Headless via headers (default — works over SSH / CI), OAuth as alternative.

  • mcp · headless claude mcp add --transport http axiom https://mcp.axiom.co/mcp --header "Authorization: Bearer $AXIOM_TOKEN" --header "x-axiom-org-id: $AXIOM_ORG_ID"
  • mcp · oauth Same command without the headers — first call opens a browser. Use on desktop where OAuth is easy.
  • tools queryApl, listDatasets, getDatasetSchema, getMonitors.
  • skills npx skills add axiomhq/skills — SRE skill = hypothesis-driven incident investigation methodology.
  • env AXIOM_TOKEN + AXIOM_ORG_ID in .envrc; same token copied once to ~/.axiom.toml for Skills.
Next investigation: user pastes reference id → agent loads SRE skill → calls mcp__axiom__queryApl with the request-timeline template → sees err.type (e.g. ExpiredError) → targeted fix.

Pino does the JSON serialization. pino.stdSerializers.err handles error objects.

Alerting — 3 monitors at launch

spike Auth failures

level=50, msg="auth.initData.failed". Alert if > 20 / 5 min.

spike Procedure errors

err.code != "NOT_FOUND". Alert if > 50 / 5 min.

regression p95 latency

p95(duration_ms) > 2000ms for 10 min.

Configured in Axiom UI after the integration is installed. Notifies email + dev webhook.

Testing

  • unit getRequestId() returns context value across awaited callbacks; err serializer keeps type / message / stack
  • integration bad Authorization header → response includes requestId + one auth.initData.failed line emitted
  • integration bot update with thrown handler error logs bot.update.unhandled exactly once
  • UAT trigger a known-auth-failure path post-deploy; confirm Axiom shows the structured event in < 30s; pnpm logs:trace <id> prints the full timeline
